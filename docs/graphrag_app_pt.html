Análise técnica detalhada

Como o app graph_rag_knowledgebase montou o seu Graph RAG

Este documento explica, em português e com base no código real do projeto, como a aplicação constrói um pipeline de recuperação jurídica que combina RAG vetorial, GraphRAG opcional, ontologia jurídica, extração de entidades e relações via LLM e um fallback por triplas persistidas quando a consulta principal do grafo falha.

Python 3.14.3 Streamlit LangChain MongoDB Atlas Vector Search MongoDBGraphStore ChatOpenAI + Voyage embeddings
Resumo curto O app não implementa um GraphRAG “do zero” com algoritmos próprios de travessia. Ele usa o MongoDBGraphStore da integração langchain-mongodb para extrair e persistir estrutura de grafo, enquanto mantém um pipeline vetorial clássico para resposta principal.
Ideia central A base jurídica é representada de duas formas ao mesmo tempo: como texto indexado semanticamente e como rede de entidades/relacionamentos. Assim, o app consegue responder por similaridade textual e também por conexões semânticas mais explícitas.
Ponto importante Na prática, a UI executa dois caminhos paralelos: Vector RAG answer e, opcionalmente, GraphRAG answer. Ou seja, o grafo complementa a aplicação; ele não substitui o fluxo vetorial.

1. Leitura geral da arquitetura

O projeto organiza o problema em quatro camadas: modelagem semântica dos dados, indexação vetorial, ingestão do grafo e orquestração da consulta em uma interface Streamlit.

1
Dataset e ontologia
Documentos jurídicos sintéticos e relações explícitas são gerados em JSONL. A ontologia jurídica em TTL define classes e relações que orientam a representação do domínio.
2
Representação vetorial
Cada documento vira um Document do LangChain e recebe embedding via Atlas Embedding API com modelo voyage-4-large.
3
Representação em grafo
Os mesmos documentos podem ser enviados ao MongoDBGraphStore, que usa LLM para extrair entidades e relações e gravar isso na coleção de grafo.
4
Consulta
A pergunta do usuário passa por busca vetorial top_k e, se habilitado, por graph_store.chat_response(question).
5
Resiliência
Se a consulta principal do GraphRAG falhar, o app usa as relações persistidas em legal_relations para montar um prompt factual de fallback.

2. Como os dados jurídicos foram modelados

A aplicação parte de um conjunto sintético pequeno, mas semanticamente bem escolhido. Em rag_knowledgebase/dataset.py, há 12 documentos cobrindo normas, dispositivos, decisões, processo, tribunal, doutrina e tópicos. Isso é importante porque o GraphRAG depende de material com entidades e vínculos identificáveis para gerar um grafo útil.

O projeto não usa dados jurídicos brutos gigantescos. Ele cria uma amostra controlada do domínio para demonstrar o mecanismo de GraphRAG com alta legibilidade.
Elemento Como aparece no projeto Função no GraphRAG
Documentos Norm, Provision, Decision, Case, Court, Doctrine e Topic Servem de base textual e também de matéria-prima para extração de entidades e relações.
Relações hasProvision, applies, cites, decidesCase, heardByCourt, aboutTopic Conectam fundamentos legais, casos, cortes e tópicos, permitindo raciocínio por ligação semântica.
Ontologia Arquivo TTL com classes e propriedades jurídicas Dá coerência conceitual ao domínio e ajuda a explicar que tipo de nó e de aresta faz sentido existir.

O arquivo dataset.py define relações concretas, por exemplo: a decisão decision_stj_resp_123456 aplica prov_cc_art_186 e prov_cc_art_927, além de citar o norm_cc_2002. Isso é um excelente padrão de GraphRAG porque cria um caminho multi-hop entre decisão, fundamento legal, norma superior, caso e tópico.

Decisão STJ -> cita -> Código Civil

Há ainda um detalhe de implementação importante: a função generate_legal_dataset(...) só escreve a ontologia mínima embutida no código se o arquivo TTL ainda não existir. Como o projeto já traz um arquivo data/legal_ontology.ttl mais rico, esse arquivo mais completo tende a ser preservado durante o uso real.

3. Como o pipeline vetorial foi montado

O caminho vetorial é a espinha dorsal da resposta principal. Ele é implementado em rag_knowledgebase/rag_pipeline.py e segue um fluxo clássico de RAG, mas com algumas escolhas relevantes para o domínio jurídico.

  1. O app cria um cliente de embeddings próprio, a classe VoyageEmbeddings, que chama a Atlas Embedding API em https://ai.mongodb.com/v1/embeddings.
  2. Cada linha do dataset é transformada em um Document do LangChain por _build_documents.
  3. O campo textual do documento não é só o texto bruto. Ele concatena Title, Type, Topic e Text. Isso injeta metadados semânticos no conteúdo indexado.
  4. O store vetorial usa MongoDBAtlasVectorSearch com similaridade por cosseno.
  5. Na consulta, o app executa similarity_search_with_score(question, k=settings.top_k) e usa os documentos retornados como contexto do LLM.
Técnica importante: enriquecimento do texto indexado Em vez de vetorizar apenas o conteúdo principal, o app inclui título, tipo e tópico no page_content. Para domínio jurídico, isso melhora a recuperação porque perguntas muitas vezes mencionam categoria documental, assunto ou base normativa, e não apenas trechos literais.
Técnica importante: metadados paralelos O mesmo documento leva um metadata estruturado com doc_id, doc_type, topic, jurisdiction, source_type, datas e número do caso. Isso prepara o terreno para filtros, inspeção de contexto e explicabilidade.
Inferência fundamentada pelo código: como o vector store é instanciado com text_key="text" e embedding_key="embedding", e recebe add_documents(docs), a coleção vetorial é usada como repositório de texto indexável + vetor de embedding + metadados associados.

4. Como o GraphRAG foi montado de fato

O GraphRAG do projeto nasce do reaproveitamento dos mesmos documentos já usados no fluxo vetorial, mas enviados para outra infraestrutura de recuperação: o MongoDBGraphStore. O app não descreve manualmente algoritmos de parsing de entidades, regras de normalização ou persistência detalhada do grafo. Em vez disso, ele delega isso à implementação da biblioteca.

  1. O código carrega dinamicamente MongoDBGraphStore de langchain_mongodb.graphrag.graph.
  2. Ele cria um ChatOpenAI com temperature=0 e o passa como entity_extraction_model.
  3. O método graph_store.add_documents(docs) recebe os documentos jurídicos e dispara a etapa de extração de entidades e relações.
  4. Em consulta, o método graph_store.chat_response(question) é usado para obter a resposta do grafo.
Em outras palavras: o projeto monta o GraphRAG por composição de componentes prontos, não por implementação artesanal do grafo. A inteligência do grafo fica dividida entre o desenho do dataset, a ontologia do domínio e a biblioteca MongoDBGraphStore, que usa LLM para estruturar o conhecimento.
Camada O que o app controla diretamente O que fica encapsulado na biblioteca
Modelagem do domínio Tipos de documento, relações JSONL e ontologia TTL Normalização final que o GraphStore faz internamente
Extração do grafo Escolha do LLM e envio dos documentos Identificação de entidades, criação de nós, arestas e formato persistido
Consulta GraphRAG Chamada a chat_response(question) Estratégia interna de recuperação e resposta com base no grafo

Há também um indício importante em db_setup.py: a coleção de grafo recebe índices para campos como entity_id, source_id, target_id e relation. Isso sugere a expectativa de uma estrutura com nós e arestas, mas o formato exato final continua dependente da biblioteca.

5. Técnicas usadas pelo app

1. Modelagem orientada por ontologia O domínio não é tratado como texto indistinto. O app parte de uma ontologia jurídica com classes e relações, o que dá sentido semântico ao grafo e ajuda a explicar por que certas conexões são válidas.
2. Dataset sintético controlado Em vez de ingerir um corpus ruidoso, o projeto cria um conjunto pequeno e coerente, excelente para demonstrar precisão estrutural, multi-hop e causalidade jurídica.
3. Embeddings semânticos especializados O uso de voyage-4-large via Atlas Embedding API dá ao app uma camada de busca semântica robusta antes mesmo de o grafo entrar em ação.
4. Graph extraction via LLM O grafo não vem todo pronto do JSONL. O app usa um LLM como modelo de extração de entidades para transformar documentos em estrutura navegável.
5. Prompting restritivo Na resposta vetorial, o sistema instrui o LLM a responder somente com base no contexto recuperado e a admitir falta de evidência quando necessário. Isso reduz alucinação.
6. Graceful degradation Quando o GraphStore principal falha, o app não abandona a consulta. Ele recorre às triplas persistidas em legal_relations para montar uma resposta factual de contingência.

6. Fluxo completo da execução

  1. O usuário abre o Streamlit e o app valida Python 3.14.3, variáveis de ambiente e configuração.
  2. Se o banco ainda não existir, o app cria coleções, índices B-Tree, Atlas Search index e Atlas Vector Search index.
  3. O usuário pode gerar o dataset jurídico e as relações explícitas a partir do botão da sidebar.
  4. Ao fazer ingestão vetorial, os documentos são convertidos em Document, embedados e gravados na coleção vetorial.
  5. Ao fazer ingestão do grafo, os mesmos documentos são enviados ao MongoDBGraphStore com um LLM de extração.
  6. Na consulta, a resposta vetorial é sempre montada primeiro com base nos documentos semanticamente mais próximos.
  7. Se a opção estiver marcada, o app também pede uma resposta ao GraphRAG.
  8. Se a consulta do grafo falhar, o app tenta responder usando as relações JSONL já persistidas em MongoDB.
  9. A interface mostra a resposta vetorial, os contextos recuperados e, separadamente, a resposta do grafo.

7. O que este app faz muito bem

  • Separa ingestão vetorial e ingestão do grafo, o que facilita demonstrar cada estágio do pipeline.
  • Usa uma modelagem jurídica plausível, com vínculos entre normas, dispositivos, decisões, processos e tópicos.
  • Mostra a diferença entre recuperar texto parecido e recuperar estrutura de conhecimento.
  • Preserva explicabilidade, porque a UI exibe contextos recuperados e metadados dos documentos usados.
  • Prevê falha operacional do GraphRAG e oferece fallback, o que é uma decisão prática muito boa para demos e PoCs.

8. Limitações e observações críticas

Esta parte é importante: o app é uma boa prova de conceito, mas ainda não é um GraphRAG jurídico de produção.
  • Não há chunking. Cada registro é indexado como um documento inteiro. Para um corpus jurídico real, isso tende a limitar granularidade e precisão da recuperação.
  • O GraphRAG não está combinado ao vetor em uma estratégia híbrida única. A UI apenas exibe as duas respostas em paralelo.
  • A função ensure_database_objects só cria índices automaticamente quando o banco ainda não existe. Se o banco existir sem índices corretos, a inicialização é pulada.
  • O fallback por relações lê apenas até 30 arestas sem filtragem por pergunta, então ele é resiliente, mas ainda é pouco seletivo.
  • O grafo depende de chamadas LLM para extração de entidades, o que traz custo, latência e possível variabilidade.

9. Conclusão

O app monta o seu GraphRAG a partir de uma ideia simples e muito eficaz para demonstração: ele pega um domínio jurídico pequeno, mas semanticamente rico, gera duas representações complementares desse domínio e consulta as duas.

A camada vetorial responde pela recuperação semântica clássica. A camada de grafo tenta estruturar conhecimento sobre entidades e vínculos legais. O resultado é uma aplicação que mostra, de forma didática, como sair de um RAG puramente textual para um RAG com noção explícita de relação jurídica.

Em resumo: este projeto não inventa um novo algoritmo de GraphRAG; ele orquestra bem componentes existentes para mostrar como um GraphRAG jurídico pode ser construído sobre MongoDB, LangChain, embeddings e LLMs.
