Detailed Technical Analysis

How the graph_rag_knowledgebase app built its Graph RAG

This document explains, based on the actual project source code, how the application builds a legal retrieval pipeline that combines vector RAG, optional GraphRAG, a legal ontology, LLM-driven entity and relation extraction, and a persisted-triples fallback when the primary graph query fails.

Python 3.14.3 Streamlit LangChain MongoDB Atlas Vector Search MongoDBGraphStore ChatOpenAI + Voyage embeddings
Short summary The app does not implement a GraphRAG "from scratch" with custom graph traversal algorithms. It uses MongoDBGraphStore from the langchain-mongodb integration to extract and persist graph structure, while maintaining a classic vector pipeline as the primary retrieval path.
Core idea The legal knowledge base is represented in two complementary ways simultaneously: as semantically indexed text and as an entity/relationship network. This allows the app to answer by textual similarity as well as by more explicit semantic connections.
Key point In practice, the UI executes two parallel paths: Vector RAG answer and, optionally, GraphRAG answer. The graph therefore complements the application — it does not replace the vector pipeline.

1. Overall architecture overview

The project organises the problem into four layers: semantic data modelling, vector indexing, graph ingestion, and query orchestration through a Streamlit interface.

1
Dataset & ontology
Synthetic legal documents and explicit relations are generated in JSONL format. A legal ontology in TTL defines classes and relations that guide the domain representation.
2
Vector representation
Each document becomes a LangChain Document and receives an embedding via the Atlas Embedding API using the voyage-4-large model.
3
Graph representation
The same documents can be sent to MongoDBGraphStore, which uses an LLM to extract entities and relations and write them to the graph collection.
4
Query
The user's question goes through a top_k vector search and, if enabled, through graph_store.chat_response(question).
5
Resilience
If the primary GraphRAG query fails, the app uses the relations already persisted in legal_relations to build a factual fallback prompt.

2. How the legal data was modelled

The application starts from a small but semantically well-chosen synthetic dataset. In rag_knowledgebase/dataset.py, there are 12 documents covering norms, provisions, decisions, proceedings, courts, doctrine, and topics. This is important because GraphRAG depends on material with identifiable entities and links to generate a useful graph.

The project does not use a massive raw legal corpus. Instead, it creates a controlled domain sample to demonstrate the GraphRAG mechanism with high readability and predictable structure.
Element How it appears in the project Role in GraphRAG
Documents Norm, Provision, Decision, Case, Court, Doctrine, and Topic Serve as both the textual base and the raw material for entity and relation extraction.
Relations hasProvision, applies, cites, decidesCase, heardByCourt, aboutTopic Connect legal grounds, cases, courts, and topics, enabling reasoning through semantic links.
Ontology TTL file with legal classes and properties Provides conceptual coherence to the domain and explains what types of nodes and edges are meaningful.

The dataset.py file defines concrete relations — for example: the decision decision_stj_resp_123456 applies prov_cc_art_186 and prov_cc_art_927, and also cites norm_cc_2002. This is an excellent GraphRAG pattern because it creates a multi-hop path across decision, legal ground, parent norm, case, and topic. 

STJ Decision  -> applies    -> Art. 186
STJ Decision  -> applies    -> Art. 927
STJ Decision  -> cites      -> Civil Code
STJ Decision  -> decidesCase -> REsp 123456
REsp 123456   -> heardByCourt -> STJ
Art. 186      -> aboutTopic  -> Moral Damages

There is also an important implementation detail: the function generate_legal_dataset(...) only writes the minimal ontology embedded in the code if the TTL file does not yet exist. Since the project already ships a richer data/legal_ontology.ttl, that more complete file tends to be preserved during actual usage.

3. How the vector pipeline was built

The vector path is the backbone of the primary response. It is implemented in rag_knowledgebase/rag_pipeline.py and follows a classic RAG flow, with a few relevant choices specific to the legal domain.

  1. The app creates its own embeddings client — the VoyageEmbeddings class — which calls the Atlas Embedding API at https://ai.mongodb.com/v1/embeddings.
  2. Each row in the dataset is converted into a LangChain Document by _build_documents.
  3. The document's text field is not just raw content. It concatenates Title, Type, Topic, and Text, injecting semantic metadata directly into the indexed content.
  4. The vector store uses MongoDBAtlasVectorSearch with cosine similarity.
  5. At query time, the app calls similarity_search_with_score(question, k=settings.top_k) and uses the returned documents as LLM context.
Key technique: enriched indexed text Rather than vectorising only the main content, the app includes the title, type, and topic in the page_content. For a legal domain, this improves retrieval because questions often reference document category, subject matter, or normative basis — not just literal text excerpts.
Key technique: parallel metadata The same document carries a structured metadata object containing doc_id, doc_type, topic, jurisdiction, source_type, dates, and case number — laying the groundwork for filtering, context inspection, and explainability.
Code-grounded inference: since the vector store is instantiated with text_key="text" and embedding_key="embedding" and receives add_documents(docs), the vector collection acts as a repository of indexable text + embedding vector + associated metadata.

4. How the GraphRAG was actually built

The project's GraphRAG is born from reusing the same documents already used in the vector pipeline, but routing them to a different retrieval infrastructure: MongoDBGraphStore. The app does not manually describe entity parsing algorithms, normalisation rules, or detailed graph persistence. Instead, it delegates all of that to the library implementation.

  1. The code dynamically loads MongoDBGraphStore from langchain_mongodb.graphrag.graph.
  2. It creates a ChatOpenAI instance with temperature=0 and passes it as the entity_extraction_model.
  3. The method graph_store.add_documents(docs) receives the legal documents and triggers the entity and relation extraction step.
  4. At query time, graph_store.chat_response(question) is called to obtain the graph-based answer.
In short: the project assembles its GraphRAG by composing ready-made components, not by hand-crafting graph algorithms. The graph intelligence is split between the dataset design, the domain ontology, and the MongoDBGraphStore library, which uses an LLM to structure knowledge.
Layer What the app controls directly What is encapsulated in the library
Domain modelling Document types, JSONL relations, and TTL ontology Final normalisation performed internally by the GraphStore
Graph extraction LLM selection and document submission Entity identification, node/edge creation, and persisted format
GraphRAG query Call to chat_response(question) Internal retrieval and graph-based response strategy

There is also an important clue in db_setup.py: the graph collection receives indexes on fields such as entity_id, source_id, target_id, and relation. This points to an expected structure with nodes and edges, though the exact final format remains dependent on the library.

5. Techniques used by the app

1. Ontology-driven modelling The domain is not treated as undifferentiated text. The app starts from a legal ontology with classes and relations, giving the graph semantic meaning and explaining why certain connections are valid.
2. Controlled synthetic dataset Rather than ingesting a noisy corpus, the project creates a small, coherent set — ideal for demonstrating structural precision, multi-hop traversal, and legal causality.
3. Specialised semantic embeddings Using voyage-4-large via the Atlas Embedding API gives the app a robust semantic search layer even before the graph comes into play.
4. LLM-driven graph extraction The graph is not pre-built from the JSONL. The app uses an LLM as an entity extraction model to transform documents into a navigable structure.
5. Restrictive prompting In the vector response, the system instructs the LLM to answer only based on the retrieved context and to acknowledge lack of evidence when necessary — reducing hallucination.
6. Graceful degradation When the primary GraphStore fails, the app does not abandon the query. It falls back to the triples persisted in legal_relations to construct a factual contingency response.

6. Full execution flow

  1. The user opens Streamlit and the app validates Python 3.14.3, environment variables, and configuration.
  2. If the database does not yet exist, the app creates collections, B-Tree indexes, an Atlas Search index, and an Atlas Vector Search index.
  3. The user can generate the legal dataset and explicit relations from the sidebar button.
  4. During vector ingestion, documents are converted to Document objects, embedded, and written to the vector collection.
  5. During graph ingestion, the same documents are sent to MongoDBGraphStore with an extraction LLM.
  6. At query time, the vector answer is always built first, based on the semantically closest documents.
  7. If the option is checked, the app also requests a GraphRAG answer.
  8. If the graph query fails, the app attempts to answer using the JSONL relations already persisted in MongoDB.
  9. The interface displays the vector answer, the retrieved contexts, and — separately — the graph answer.

7. What this app does very well

  • Separates vector ingestion from graph ingestion, making it easy to demonstrate each pipeline stage independently.
  • Uses a plausible legal model with links between norms, provisions, decisions, proceedings, and topics.
  • Illustrates the difference between retrieving similar text and retrieving structured knowledge.
  • Preserves explainability — the UI exposes retrieved contexts and document metadata.
  • Anticipates GraphRAG operational failure and provides a fallback, which is a very practical decision for demos and PoCs.

8. Limitations and critical observations

This part matters: the app is a solid proof of concept, but it is not yet a production-grade legal GraphRAG.
  • There is no chunking. Each record is indexed as a whole document. For a real legal corpus, this tends to limit retrieval granularity and precision.
  • GraphRAG is not combined with the vector path into a single hybrid strategy. The UI simply displays both answers side by side.
  • The ensure_database_objects function creates indexes automatically only when the database does not yet exist. If the database already exists without the correct indexes, initialisation is skipped.
  • The fallback via relations reads up to 30 edges without filtering by question, so it is resilient but still not very selective.
  • The graph relies on LLM calls for entity extraction, which introduces cost, latency, and possible variability in results.

9. Conclusion

The app builds its GraphRAG from a simple yet highly effective demonstration idea: take a small but semantically rich legal domain, generate two complementary representations of that domain, and query both.

The vector layer handles classic semantic retrieval. The graph layer attempts to structure knowledge about legal entities and relationships. The result is an application that shows, in a didactic way, how to move from a purely textual RAG to a RAG with explicit awareness of legal relationships.

In summary: this project does not invent a new GraphRAG algorithm — it orchestrates existing components effectively to show how a legal GraphRAG can be built on top of MongoDB, LangChain, embeddings, and LLMs.
